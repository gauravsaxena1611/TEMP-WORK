# [](#component-diagram)Component diagram

Next you can zoom in and decompose a container to describe the [components](/abstractions/component) that reside inside it; including their responsibilities and the technology/implementation details.

## [](#scope)Scope

A single container.

## [](#primary-elements)Primary elements

Components within the container in scope.

## [](#supporting-elements)Supporting elements

Containers (within the software system in scope) plus people and software systems directly connected to the components.

## [](#intended-audience)Intended audience

Software architects and developers.

## [](#recommended)Recommended?

No, only create component diagrams if you feel they add value, and consider automating their creation for long-lived documentation.
